# ORGS evaluation — 2026-04-19

Verdict: **ITERATE** (round 1 of 2 budgeted)

## Epic goal, restated

Provide `orgs_add` and `orgs_update` RPCs so new forge orgs can be
onboarded and mutated without hand-editing `~/.config/hyperforge/orgs/`.

## Supports

Enables `foyer` (and every future project under a new forge org such as
`juggernautlabs`) to be bootstrapped with a single script rather than a
YAML hand-edit.

## Actions — bounded tightening list

1. **ORGS-2 §Required behavior:** replace "(illustrative)" with a pinned
   definition. Name the struct `OrgSshKeys`, place it at
   `src/hub_params.rs` (new file) or `src/config/org.rs` (planner's
   call — whichever is picked, name it in the ticket). Pin the derive
   set: `#[derive(Deserialize, JsonSchema, Debug, Default)]`. The
   `JsonSchema` derive is not optional — without it, Synapse cannot
   generate the per-field flags the acceptance criteria depend on.

2. **ORGS-2 §Required behavior table:** replace every "Error event" /
   "Info event" cell with the specific `HyperforgeEvent` variant and
   its expected payload fields (e.g. `HyperforgeEvent::Error { code:
   ErrorCode::OrgExists, message: String }`). If the exact variant
   list needs to be chosen by the implementer, write a half-ticket
   spike to pin it before promotion.

3. **ORGS-3 §Required behavior table:** add a row for *"flag absent
   (not passed at all)"* alongside the existing *"present, empty
   string"* and *"present, non-empty"* rows. Make explicit that an
   absent flag leaves the on-disk value untouched.

4. **ORGS-3 §Required behavior:** apply the same event-variant pinning
   from action 2.

## Re-evaluation trigger

re-evaluation still produces ITERATE with the same items, the

ITERATE, verdict flips to RE-SHAPE (the goal likely needs re-examining).
